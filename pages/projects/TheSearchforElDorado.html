The Search for El Dorado

A tale from nothing to the wonders of a full fledged website

With GitHub as the world that serves as landscape for static hosting I started my journey towards a better identity on the web. I have tried different sets of development tools to accomplish my mission, in this post I will outline what did I use and the reasons for me to do it.

First of all GitHub is known as the tool for excellence in open source collaboration, it has nice features amongst those the capability of hosting static sites known as GitHub Pages.

Figure 1. GitHub Pages

GitHub pages can make use of custom domains to customize even more your experience. So it seems like we have a good start for our journey.

Starting into the wide world of the web we get tons and tons of frameworks, many of them of very high quality and with all sorts of advantages. After all everyone try to solve their own problems and eventually reach to a solution that can be used by others, add on top of that open source collaboration and we get really good quality frameworks and libraries. As in nature the better adapted is the one who survives, same here only those frameworks that are developed with the right practices, solves problems in the most efficient way and with a good pace (web development is of course quite agile and changes a lot) are the ones that survive. Sometimes these frameworks are adopted by companies (or created by them) because it solves problems for them and so they get supported too. Now we get more interests involved, and while it sounds good because having a company making a bet on a framework speaks of quality, reliability, and support; it can also be bad as they try to influence their design and to push the frameworks to match their products (making them more likely to buy too). So it requires a good balance. And so the reason for this preamble is the selection of the right framework which took me some time to master, and then re-master again. As a person I placed my bet too in a framework that could be reliable, help me solve most of my problems; I needed this because I cannot be learning every possible pattern and implementation out there.

JavaScript

Oh JavaScript a long friend with hidden caveats all over, proves to be friend-(ly) yet it comes and bites you if you’re not careful. But being on the web requires JavaScript, several people have had enough of its nuances so potential solutions were also developed. TypeScript (amongst many other tools for making your life easier with JavaScript) seemed to me the best offer in terms of adoption, maturity, ease of use as well as being supported by two different companies (Google and Microsoft) which tells me quality, support, and long term commitment from all parties. ‘In an insane work it was the sanest choice’. TypeScript lines up naturally with the current version of Angular (Angular 4 as of the time of creating this post - which was re-designed to adopt TypeScript since version 2).

Figure 2 - TypeScript Language

Angular solves several problems, in particular provides better interaction with the DOM (2-way), a more formal structure allows for separation of concerns, security, dependency injection, testability and so many additional goodies. All of these are good enough reason to try it out. Which I did in this site. I ended up loving the framework and the philosophy behind it and this kept me continuing developing my site in this framework. We have a horse!

Figure 3 - Angular Framework

User Interface

While the best way for representing yourself in a unique way is to design the user interface from the ground up, this is not achievable unless you have a good amount of resources yourself (time, knowledge, money). So I relied on additional tooling for UI, designs, components, icons, fonts, etcetera. The natural choice for me due to its ease of implementation and use was Bootstrap. It allowed me to get something running faster and with better design. It has been flexible enough for my needs so I used it throughout my site and so we have a map!

On to the adventure! There are still plenty in my backlog to make my site look, feel, and work the way I want, but now I’m in a position to start adding content to the site.

Figure 4 - Bootstrap Framework

Content

With the previous set of tools I can now create a shell that feels and looks similar to what I want, however the content was always an interesting set of questions. Will I end up writing HTML? How about boxing myself into a better authoring experience like those of Medium? How about an intermediate point and write markup to ease the pain of plain HTML like SlackEdit? The old Windows Live Writer? Any WYSIWYG editor?

All of these were options at some point, however the latest and best tool for me ended up being Google Docs where I can author my content the way I want and with a great range of tools for authoring content. To publish (for now) all I do is export the doc as HTML and publish it on my site. Done! We got weapons!!!

Figure 5 - Google Docs

Wrapping Up

I will try the aforementioned tools to continue building this site as well as to share my learning, more technical details about it and in general other areas of my interest in the content. Feel free to add your suggestions and comments.
